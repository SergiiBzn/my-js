### **Software-Design-Prinzipien: SOLID, KISS, DRY, YAGNI**  

üìå **Diese Prinzipien helfen, sauberen, wartbaren und effizienten Code zu schreiben.**  
Sie verbessern die **Lesbarkeit, Skalierbarkeit und Testbarkeit** von Code.

## **1. SOLID ‚Äì F√ºnf Prinzipien f√ºr objektorientierte Programmierung**
üìå **SOLID steht f√ºr f√ºnf Prinzipien, die den Code modular, flexibel und erweiterbar machen.**

| Prinzip | Beschreibung | Beispiel |
|---------|-------------|----------|
| **S ‚Äì Single Responsibility** | **Jede Klasse/Funktion sollte nur eine Aufgabe haben.** | ‚ùå **Schlecht:** Eine Klasse, die **Daten speichert und anzeigt.**<br>‚úÖ **Gut:** Eine Klasse speichert Daten, eine andere zeigt sie an. |
| **O ‚Äì Open/Closed Principle** | **Code sollte erweiterbar, aber nicht √§nderbar sein.** | ‚úÖ **Gut:** Neue Features √ºber **Vererbung oder Interfaces** hinzuf√ºgen, nicht alten Code √§ndern. |
| **L ‚Äì Liskov Substitution Principle** | **Subklassen m√ºssen sich wie ihre Elternklasse verhalten.** | ‚ùå **Schlecht:** Eine `Rechteck`-Klasse wird zu `Quadrat`, aber ver√§ndert unerwartet Verhalten. |
| **I ‚Äì Interface Segregation** | **Keine riesigen Interfaces ‚Äì lieber mehrere kleine!** | ‚ùå **Schlecht:** Ein Interface mit 10 Methoden, von denen nur 2 verwendet werden. |
| **D ‚Äì Dependency Inversion** | **Hochrangige Module sollen nicht von konkreten Implementierungen abh√§ngen.** | ‚úÖ **Gut:** Nutzung von **Abstraktionen (`interface`)**, nicht direkt von Klassen. |

Hier sind Beispiele f√ºr alle **SOLID**-Prinzipien in JavaScript:

### **1. S ‚Äì Single Responsibility Principle (SRP)**
üìå **Jede Klasse sollte nur eine Verantwortung haben.**  
‚úÖ **Logging (`FileLogger`) und Benutzerverwaltung (`User`) sind getrennt.**

### **2. O ‚Äì Open/Closed Principle (OCP)**
üìå **Klassen sollten offen f√ºr Erweiterungen, aber geschlossen f√ºr Modifikationen sein.**  
‚úÖ **Neue Logging-Methoden (`FileLogger` ‚Üí `DatabaseLogger`) werden hinzugef√ºgt, ohne `User` zu √§ndern.**

  }

  }

  constructor(name, logger) {
    this.name = name;
    this.logger = logger;
  }

    this.logger.log(`Benutzer ${this.name} gespeichert.`);

### **3. L ‚Äì Liskov Substitution Principle (LSP)**
üìå **Unterklassen sollten die Elternklasse korrekt ersetzen k√∂nnen.**  
‚úÖ **Das Ersetzen von `Rectangle` durch `Square` zerst√∂rt nicht die Logik.**

### **4. I ‚Äì Interface Segregation Principle (ISP)**
üìå **Klassen sollten nicht von Methoden abh√§ngen, die sie nicht ben√∂tigen.**  
‚úÖ **`Printer` und `Scanner` wurden getrennt, sodass ein Drucker nicht `scan()` ben√∂tigt.**

### **5. D ‚Äì Dependency Inversion Principle (DIP)**
üìå **Abh√§ngigkeiten sollten von Abstraktionen (`interface`) und nicht von konkreten Klassen kommen.**

  }

  }

    this.logger.log(`Benutzer ${this.name} gespeichert.`);
  }

## **3. DRY ‚Äì "Don't Repeat Yourself"**
üìå **Vermeide duplizierten Code ‚Äì nutze Funktionen oder Klassen f√ºr Wiederverwendbarkeit.**  

‚ùå **Schlechtes Beispiel (duplizierter Code)**

‚úÖ **Wiederverwendbar & flexibel**
